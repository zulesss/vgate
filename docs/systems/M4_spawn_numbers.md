## §1 Spawn Ramp

### Проблема исходной формулы

PLAN предлагает `interval = 4 / (1 + t*0.015)` (Devil Daggers форма). Проверяем:

| t (сек) | interval | накоплено врагов (est.) |
|---------|----------|------------------------|
| 0       | 4.0 сек  | —                      |
| 60      | 2.1 сек  | ~30                    |
| 120     | 1.43 сек | ~65                    |
| 180     | 1.08 сек | ~105                   |
| 193     | ~1.0 сек | ~115                   |
| 600     | 0.4 сек  | ~400+ (с cap — нет)    |

**Проблема**: пик давления (interval ~1 сек) достигается на t≈193 сек — это **3.2 минуты**, а не середина 10-15-минутного run'а. После 3 минут нарастание по сути прекращается (от 1.0 до 0.4 сек разница в feel незначительна), а run продолжается ещё 7-12 минут. Нет нарастания — нет "вот-вот сломаюсь" в финале.

В Devil Daggers run длится 1-5 минут. Для 10-15 мин нужна другая кривая.

### Рекомендуемая формула

```
interval = 4.0 / (1 + t * 0.005)
```

| t (сек) | interval | сравнение с оригиналом |
|---------|----------|------------------------|
| 0       | 4.0 сек  | = 4.0 сек              |
| 60      | 3.23 сек | было 2.1 сек           |
| 120     | 2.78 сек | было 1.43 сек          |
| 180     | 2.43 сек | было 1.08 сек          |
| 300     | 2.0 сек  | было 0.74 сек          |
| 480     | 1.43 сек | было 0.51 сек          |
| 600     | 1.21 сек | было 0.4 сек           |
| 720     | 1.05 сек | —                      |
| 800     | 0.95 сек | — (пик давления)       |

Пик (~1 сек interval) достигается на t≈800 сек = **13.3 минуты** — середина финала run'а. Это и есть "вот-вот сломаюсь".

**Почему 0.005, а не 0.015**: коэффициент k=0.015 взят из Devil Daggers где run ~3 мин. Для 10-15 мин нужен k ≈ 0.015/3 = 0.005. Пропорция прямая.

### Floor (минимальный интервал)

Вводим `interval_min = 0.8` сек. Без floor при t=1000+ interval уходит в 0.36 сек — это burst 1-2 врага в секунду, performance killer и visual noise. 0.8 сек = максимальный темп, при котором арена визуально читаема.

**Итоговая формула:**
```
interval = max(0.8, 4.0 / (1 + t * 0.005))
```

### Enemy cap на арене

PLAN: «60 FPS при 30+ врагах». Исходя из этого:

- **Рекомендуемый cap = 25**. Обоснование: при 30+ start'ует performance risk. 25 — буфер под 30 с учётом, что все 25 могут атаковать одновременно. Если playtest покажет что арена слишком пустая — поднять до 30. Риск в обратную сторону (30 → переполнение) сложнее откатить.
- При cap=25 и interval=0.8 сек поток спавна упирается в cap примерно через 20 сек после достижения floor. После этого новый враг спавнится только при смерти старого — это честный steady state.

| Параметр          | Значение |
|-------------------|----------|
| interval_min      | 0.8 сек  |
| interval_max      | 4.0 сек  |
| k (коэффициент)   | 0.005    |
| enemy_cap         | 25       |

## §2 Type Mix Curve

### Проблема

M3_identity §4: «5+ стрелков одновременно — cap drain накапливается быстрее, чем Kill verb успевает восстанавливать». Shooter penalty=10, cooldown=4 сек. При 5 стрелках: потенциальный drain = 5×10 = 50 cap за 4 сек = -12.5 cap/сек. Kill restore = +25. Чтобы компенсировать 5 стрелков нужно убивать 2 врага в 4 сек. При нарастающей арене это нереально — игрок теряет agency.

Hard rule из M3_identity: **max 4 стрелка одновременно живых на арене**.

### Кривая bias по времени

Bias реализуется как вероятность spawn type при каждом интервале:

| t (сек) | P(melee) | P(shooter) | Обоснование |
|---------|----------|------------|-------------|
| 0       | 90%      | 10%        | Игрок учится hook'у через ближнебоя. 10% стрелков — ранний введение типа без overwhelm |
| 60      | 85%      | 15%        | Стрелки начинают появляться стабильнее |
| 120     | 80%      | 20%        | Первые 2 мин: ближнебой доминирует |
| 180     | 75%      | 25%        | Конец «melee phase» по M3_identity |
| 300     | 65%      | 35%        | Стрелки как регулярная угроза |
| 480     | 55%      | 45%        | Паритет приближается, давление смешанное |
| 600+    | 50%      | 50%        | Финал: равный микс |

**Почему 90/10 на старте, а не 80/20**: melee penalty=20 vs shooter penalty=10. Два меле-хита = смерть из cap=80 без kills. Игрок в первые 30 сек ещё не знает hook. 90/10 даёт ему время понять "убей ближнебоя = восстанови cap" до того как появится позиционная угроза стрелков.

**Линейная интерполяция** между точками достаточна. Нелинейная кривая даст незаметный для игрока эффект, но усложнит debug.

Формула для кода:
```
shooter_probability = lerp(0.10, 0.50, clamp(t / 600.0, 0.0, 1.0))
```
Это точно соответствует таблице (t=0 → 10%, t=600 → 50%, линейно).

### Hard cap на стрелков

Независимо от кривой: если `live_shooters >= 4`, следующий spawn принудительно melee. Это не визуальная деталь — это защита от agency collapse.

| Параметр              | Значение |
|-----------------------|----------|
| shooter_prob t=0      | 10%      |
| shooter_prob t=600    | 50%      |
| shooter_prob curve    | linear lerp по clamp(t/600, 0, 1) |
| max live shooters     | 4        |

## §3 Score

### Требования

- Read'ается: игрок понимает "прогрессирую" между run'ами
- Multiplier за время выживания — центральный стимул
- Bonus за "in-form play" при высоком cap

### Базовый score per kill

| Тип врага | Base score | Обоснование |
|-----------|-----------|-------------|
| Melee     | 100       | Более рисковый kill (proximity, penalty=20). Доступный enemy — ценится меньше стрелка |
| Shooter   | 150       | Позиционный kill требует больше усилий. M3_identity: стрелок = intellectual challenge |

### Multiplier за время выживания

```
time_multiplier = 1.0 + (t / 300.0)
```

| t (сек) | multiplier |
|---------|-----------|
| 0       | ×1.0      |
| 60      | ×1.2      |
| 120     | ×1.4      |
| 180     | ×1.6      |
| 300     | ×2.0      |
| 480     | ×2.6      |
| 600     | ×3.0      |
| 900     | ×4.0      |

**Почему линейный multiplier, не экспоненциальный**: экспонента ломает поздний баланс (принцип из системного дизайна). При экспонентиальном multiplier kill на минуте 12 стоит в 10 раз дороже kill'а на минуте 1 — игрок кемперует. Линейный +1/300сек = +0.2 каждые минуту = читаемо и не провоцирует degenerate strategy.

### In-form bonus

Bonus при `velocity_cap >= 80` (старт cap = 80, значит "не получал хитов или восстановился полностью"):
```
bonus_multiplier = 1.25 при cap >= 80, иначе 1.0
```

**Почему 1.25, не 1.5 или 2.0**: bonus не должен перевешивать time_multiplier как основной score-driver. 25% bonus заметен в итоговом счёте (разница в несколько тысяч очков за run), но не делает "кемп с высоким cap" оптимальнее чем "убивай быстро и выживай долго".

Итоговая формула:
```
score_per_kill = base_score * time_multiplier * bonus_multiplier
```

Пример run (300 сек, 60 kills, 50% melee / 50% shooter, cap >= 80 весь run):
```
melee: 30 × 100 × 2.0 × 1.25 = 7500
shooter: 30 × 150 × 2.0 × 1.25 = 11250
total = 18750
```

Пример новичок (120 сек, 20 kills, 90% melee, частые хиты → cap редко >= 80):
```
melee: 18 × 100 × 1.4 × 1.0 = 2520
shooter: 2 × 150 × 1.4 × 1.0 = 420
total ≈ 2940
```

Разница 6× — читаемая прогрессия между забегами, не пропасть.

| Параметр              | Значение |
|-----------------------|----------|
| base_score melee      | 100      |
| base_score shooter    | 150      |
| time_multiplier       | 1 + t/300 |
| in-form bonus         | ×1.25 при cap ≥ 80 |

## §4 Restart Loop Timing

PLAN §M2 фиксирует: `1.8 сек смерть + 0.6 сек чёрный + score + 0.4 сек fade-in = 2.8 сек`.

### Оценка

**2.8 сек — правильно**, при одном условии: score screen не блокирует transition. Структура:

```
0.0с: player_died.emit → freeze player, stop spawning
0.0–1.8с: death animation / ragdoll / camera drift (feel-engineer territory)
1.8–2.4с: черный экран (0.6 сек)
2.4–2.4с: score screen появляется НА чёрном, одновременно с fade-in в фон
2.4–2.8с: fade-in арены (0.4 сек) — score overlay остаётся
2.8с: арена видна, score на экране. Кнопка [RESTART] / [QUIT] активна.
```

Игрок сам нажимает рестарт — нет принудительного ожидания. 2.8 сек — время до того как появится арена и кнопка. Score читается пока игрок двигает руку к кнопке.

**Риск "слишком быстро"**: если score экран появляется одновременно с рестартом без паузы — игрок не успевает прочитать результат. Решение: кнопка [RESTART] активируется через 0.5 сек после появления score (anti-accidental-press). Итого: жёсткая пауза = 2.8 + 0.5 сек обязательного чтения = 3.3 сек минимум перед возможностью рестарта. Это в рамках "< 3 сек death → restart" по exit criteria PLAN'а (рестарт начинается в 3.3 сек, арена загружается ещё ~0.3 сек → игрок в игре на ~3.6 сек). Укладывается в "< 5 сек" — субъективно быстро.

**Риск "слишком медленно"**: при 10-15 мин run'е игрок умирает 1-3 раза. 3-5 сек смерть на 3 смерти = 10-15 сек потеряно. Это приемлемо.

| Фаза               | Длительность | Действие |
|--------------------|-------------|----------|
| Death animation    | 1.8 сек     | camera drift, freeze |
| Черный экран       | 0.6 сек     | — |
| Score появляется   | 0.0 сек     | мгновенно при fade-in start |
| Fade-in арены      | 0.4 сек     | арена проявляется под score overlay |
| Кнопка активна     | +0.5 сек    | anti-accidental-press delay |
| **Минимум до рестарта** | **3.3 сек** | |

## §5 Spawn Safety / Pacing

### Минимальный interval между спавнами (anti-burst)

Floor уже установлен в §1: `interval_min = 0.8 сек`. Это абсолютный минимум между любыми двумя спавнами. При t→∞ формула даёт 0.4 сек без floor — это burst, визуально неприятный и нечитаемый.

**Дополнительно**: при спавне нового врага проверять что арена не переполнена (enemy_count < cap). Если cap достигнут — спавн пропускается (не накапливается в очередь), следующая попытка через interval.

### Минимальная дистанция от игрока при спавне

Арена 40×40 units. Spawn-points S1-S4 по периметру (corners / edge midpoints).

| Параметр                | Значение | Обоснование |
|------------------------|---------|-------------|
| min_spawn_distance     | 12 units | Radius detection у врагов = 35 units, но первый contact не должен быть мгновенным. 12 units при скорости melee=5.5 u/s → 2.2 сек до contact. Достаточно для реакции. |
| hard_min               | 8 units  | Fallback если все 4 spawn-points ≤ 12 units от игрока (редко, но возможно при кемпинге угла). |

Если игрок находится ≤ 12 units от spawn-point — выбрать другой. Если все ≤ 12 units — spawn на дальнейшей точке (даже если ≤ 8 units).

### Cooldown на spawn-point

Один spawn-point не спавнит чаще чем 1 раз в N секунд, даже если global interval позволяет.

| Параметр                 | Значение | Обоснование |
|--------------------------|---------|-------------|
| spawn_point_cooldown     | 3.0 сек  | При 4 spawn-points и interval=0.8 сек: в худшем случае одна точка спавнит каждые 3.2 сек — cooldown 3.0 сек не блокирует ничего. При interval=1.5 сек: точка используется каждые 6 сек — cooldown не ощущается. Защита только от edge case когда алгоритм выбора точки "залипает" на одной. |

Алгоритм выбора spawn-point: выбрать случайную точку среди тех, что (a) не на cooldown и (b) > min_spawn_distance от игрока. Если таких нет — выбрать с минимальной дистанцией среди не-на-cooldown.
